## 1. 文档目的

本文档是 `AhaKey Studio for macOS` 的首版交互蓝图，用于把产品总纲进一步收敛成可审批、可设计、可开发的 UX 规范。

本文档主要回答以下问题：

- 首屏到底长什么样
- 用户先看到什么，再点什么
- 1:1 键盘热区怎么定义
- 每个部件点开后右侧 Inspector 显示什么
- `Mode 0 / 1 / 2` 如何在一个界面里切换
- 哪些功能本次就做，哪些明确显示为“开发中”

当前已落地的首版代码范围：

- 旧 `TabView` 已被桌面优先的 `Studio` 主界面替换
- 首屏改成了左侧 1:1 键盘画布 + 右侧 Inspector
- `Mode 0 / 1 / 2`、语音预设、快捷键编辑、同步到键盘主链路都已经接进 SwiftUI 壳子
- OLED 上传、完整灯效编辑、宏编辑仍属于下一阶段

## 2. 审批范围

本轮审批只聚焦以下事项：

1. 首屏结构是否正确
2. 1:1 键盘画布是否足够直观
3. `Mode 0` 的默认心智是否准确
4. `Key 1` 的语音软件选择方式是否合理
5. `OLED` 的“只做动图，其他显示开发中”是否清晰
6. 右侧 Inspector 的结构是否适合作为首版编辑器
7. 灯条状态预览方式是否满足主流程

本轮不审批以下事项：

- 最终视觉皮肤
- 品牌配色的精修
- 动效时长的精调
- 所有后续扩展功能的最终交互

---

## 3. 首版主界面结论

### 3.1 单窗口结构

首版主界面采用单窗口三段式结构：

1. 顶部工具栏
2. 中央数字孪生画布
3. 右侧 Inspector 检查器

不采用：

- 多标签页主导航
- “设备 / 键位 / 屏幕”分离式设置页
- 首屏大列表或表格

### 3.2 首屏默认着陆点

应用启动并连接设备后，默认进入：

`Mode 0 + Key 1 已选中`

原因：

- `Mode 0` 是官方默认 vibecoding 模式
- `Key 1` 是用户最容易理解、最有价值的入口
- 语音预设能最快证明软件价值

### 3.3 用户第一眼应该理解到的内容

首屏必须让用户在 5 秒内理解：

- 中间这就是我的键盘
- 顶部可以切换 `Mode 0 / 1 / 2`
- 点键盘上的任何部件都能改
- 改完右上角或顶部有一个明确的“同步到键盘”

### 3.4 桌面首屏硬性要求

在标准桌面窗口尺寸下，用户必须无需拖动画布、无需滚动画布，就能一次完整看到整把键盘。

因此首版实现必须满足：

- 键盘完整显示在主画布中
- Inspector 固定在右侧
- 画布不作为首屏滚动容器

只有在极小窗口尺寸下，才允许进入自适应压缩或次级滚动策略。

## 4. 页面骨架

### 4.1 页面布局

```text
┌ Toolbar ─────────────────────────────────────────────────────────────────────┐
│ AhaKey Studio | 已连接 | 84% | Mode 0 / 1 / 2 | 同步到键盘 | 更多          │
├───────────────────────────────┬─────────────────────────────────────────────┤
│ 数字孪生画布                  │ Inspector                                   │
│                               │                                             │
│  [1:1 键盘]                   │ 选中部件标题                               │
│  - 灯条                        │ 当前模式说明                               │
│  - OLED                       │ 相关配置项                                 │
│  - Key1~Key4                  │ 预设 / 描述 / 状态                         │
│  - 拨杆                        │ 开发中提示 / 高级项                        │
│                               │                                             │
├───────────────────────────────┴─────────────────────────────────────────────┤
│ Status Bar: 最近同步时间 | 当前选中对象 | 未同步改动数 | 设备反馈状态         │
└──────────────────────────────────────────────────────────────────────────────┘
```

桌面版强约束：

- 左侧数字孪生画布优先保证完整可见
- 右侧 Inspector 保持固定存在
- 不允许在桌面默认宽度下掉到“画布在上、Inspector 在下”的移动端式布局

### 4.2 工具栏控件清单

- App 名称：`AhaKey Studio`
- 设备状态：`已连接 / 连接中 / 断开`
- 电量
- 模式切换器：`Mode 0 / Mode 1 / Mode 2`
- 主按钮：`同步到键盘`
- 次按钮：`撤销`、`重做`
- 更多菜单：`导入 / 导出 / 恢复默认 / 查看帮助`

### 4.3 状态栏信息清单

- 当前选中对象
- 当前设备名
- 未同步改动数
- 最近一次同步结果
- 若有开发中功能被点击，显示轻量提示

## 5. 数字孪生画布规范

### 5.1 坐标系定义

画布使用键盘实体相对尺寸作为统一坐标系：

- 宽度：`109`
- 高度：`54`
- 单位：相对毫米坐标

设计和实现阶段均以这一坐标系为准，再按 UI 画布缩放。

### 5.2 热区定义

以下坐标为首版热区基线，不要求像 CAD 一样绝对精确，但要求视觉与点击体验一致。

| 部件 ID | 名称 | x | y | w | h | 备注 |
|------|------|------|------|------|------|------|
| `lightBar` | 灯条 | 11 | 7 | 58 | 5 | 命中区域应略大于视觉条形 |
| `oledDisplay` | OLED | 73 | 10 | 28 | 15 | 可视区域 |
| `key1` | 语音键 | 9 | 28 | 18 | 18 | 视觉主体是 `15 x 15` 键帽，点击区略放大 |
| `key2` | 确认键 | 27 | 28 | 18 | 18 | 同上 |
| `key3` | 取消键 | 45 | 28 | 18 | 18 | 同上 |
| `key4` | 回车键 | 63 | 28 | 18 | 18 | 同上 |
| `toggleSwitch` | 拨杆 | 88 | 34 | 10 | 14 | 点击区略大于底座 |

### 5.3 热区视觉状态

每个热区都必须支持以下状态：

- `default`
- `hover`
- `selected`
- `dirty`
- `syncing`
- `synced`
- `disabled`

状态表现建议：

- `hover`：浅蓝或浅青描边
- `selected`：明显描边 + 柔和外发光
- `dirty`：右上角小橙点
- `syncing`：轻微脉冲
- `synced`：短暂绿色勾形反馈
- `disabled`：半透明 + “开发中”

### 5.4 画布中的摘要信息

默认情况下，画布不是纯静态图，而是要显示极轻量摘要：

- `Key 1`：当前语音软件名，例如 `macOS 语音`
- `Key 2`：当前动作名，例如 `批准`
- `Key 3`：当前动作名，例如 `拒绝`
- `Key 4`：当前动作名，例如 `回车`
- `OLED`：当前模式下显示策略，例如 `动图`
- `Light Bar`：当前状态预览，例如 `AI 运行中`
- `Toggle Switch`：当前语义，例如 `自动批准`

摘要只做一行，不做复杂说明。

## 6. Inspector 结构

### 6.1 通用结构

Inspector 从上到下采用固定骨架：

1. 标题区
2. 当前模式说明
3. 核心配置区
4. 预览区
5. 高级配置区
6. 开发中提示区

### 6.2 选中 Key 1 时

标题：

`Key 1 · 语音键`

核心配置区：

- 语音方案选择器
- 当前生效快捷键摘要
- OLED 显示文案

语音方案选择器选项：

- `macOS 原生语音转文本`
- `Typeless / Fn`
- `微信语音`
- `Claude Code 语音快捷键`（开发中）
- `Codex 语音快捷键`（开发中）
- `豆包输入法内测版`（开发中）
- `自定义快捷键`

交互原则：

- 主路径显示“软件”
- 只有展开高级项时，才显示解析后的快捷键

### 6.3 选中 Key 2 时

标题：

`Key 2 · 确认键`

核心配置区：

- 动作类型：`标准快捷键 / 设备宏 / 主机动作`
- 官方预设：`批准`
- 当前绑定摘要
- OLED 文案

### 6.4 选中 Key 3 时

标题：

`Key 3 · 取消键`

核心配置区：

- 动作类型
- 官方预设：`拒绝`
- 当前绑定摘要
- OLED 文案

### 6.5 选中 Key 4 时

标题：

`Key 4 · 回车键`

核心配置区：

- 动作类型
- 官方预设：`回车`
- 当前绑定摘要
- OLED 文案

### 6.6 选中 OLED 时

标题：

`OLED 屏幕`

首版核心配置：

- 当前模式显示类型：`动图`
- 上传 GIF
- 预览帧数
- 帧率选择

开发中区必须出现：

- `文字显示：开发中`
- `Token 用量：开发中`
- `模型环境：开发中`

这样用户不会误以为这些功能已经完成。

### 6.7 选中 Light Bar 时

标题：

`灯条状态`

核心配置：

- 当前模式：`Mode 0 / 1 / 2`
- 当前审批模式：`自动批准 / 手动批准`
- AI 状态预览器
- 状态矩阵列表

首版状态预览器至少支持：

- `SessionStart`
- `PreToolUse`
- `PermissionRequest`
- `PostToolUse`
- `Stop`
- `TaskCompleted`
- `SessionEnd`

每次切换预览状态时，画布中的灯条立即跟着变化。

### 6.8 选中 Toggle Switch 时

标题：

`拨杆`

核心配置：

- 往上：`自动批准`
- 往下：`手动批准`
- 模式说明文案
- 自动批准对灯效的覆盖说明

说明文案必须解释清楚：

- 自动批准会覆盖部分灯条显示
- 该拨杆语义只在 `Mode 0` 的 IDE 联动场景中最重要

## 7. Mode 行为规范

### 7.1 Mode 0

定位：

- 官方默认编程模式
- 优先服务 Claude、Cursor、Codex 的 vibecoding 流程

首版默认摘要：

- Key 1：`macOS 语音`
- Key 2：`批准`
- Key 3：`拒绝`
- Key 4：`回车`
- OLED：`动图`
- 灯条：`AI 状态`

### 7.2 Mode 1

定位：

- 用户自定义模式

首版行为：

- 不强塞官方流程
- 保留键位、灯效、OLED 的自由配置入口

### 7.3 Mode 2

定位：

- 用户自定义模式

首版行为：

- 与 `Mode 1` 一致
- 作为另一套独立配置空间

## 8. 关键交互流

### 8.1 主流程：改 Key 1

1. 用户打开应用
2. 连接设备成功
3. 默认进入 `Mode 0`
4. 默认选中 `Key 1`
5. 在 Inspector 中切换语音方案
6. 画布中 `Key 1` 右上角出现未同步标记
7. 用户点击 `同步到键盘`
8. 状态栏显示同步结果
9. 画布未同步标记消失

### 8.2 次流程：改灯条预览

1. 用户点击灯条
2. Inspector 切到灯条状态面板
3. 用户切换 `PreToolUse / Stop / PermissionRequest`
4. 画布灯条即时预览
5. 若修改的是默认映射，标记未同步
6. 同步后设备获得新映射

### 8.3 OLED 流程

1. 用户点击 OLED
2. Inspector 显示 GIF 预览面板
3. 用户添加 GIF
4. 看到帧数和帧率
5. 文字、token、模型显示区域明确写 `开发中`
6. 用户同步到设备

## 10. 审批检查清单

请按以下问题审阅：

1. 首屏是否足够像真实硬件，而不是传统设置软件
2. `Mode 0` 默认进入 `Key 1` 是否合理
3. 右侧 Inspector 是否足够直接
4. `Key 1` 的软件选择方式是否比直接填快捷键更符合用户心智
5. OLED 的“开发中”提示方式是否足够清晰
6. 灯条预览是否应该放在 Inspector，而不是独立页面
7. `Mode 1 / 2` 作为纯自定义模式是否合适
8. 顶部工具栏是否简洁，不像设置中心

审批结论建议按三档记录：

- `通过`
- `通过，但有修改意见`
- `不通过，需要重做结构`
